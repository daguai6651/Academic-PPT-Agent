# 学术论文汇报智能体 — 技术方案

> **项目名称：** Academic PPT Agent（暂定名）
> **项目定位：** 面向计算机专业研究生的学术论文汇报 PPT 生成与交互式局部修改工具
> **创建日期：** 2026-04-22
> **版本：** v0.1

---

## 一、项目背景与目标

### 1.1 痛点

现有 AI PPT 生成工具的核心问题：

1. **全局修改，无法局部微调**——用户说"把这页某个图标换掉"，AI 把整页甚至整份 PPT 重新生成，越改越差
2. **生成后难以修改**——导出为 PPTX 后，AI 生成的内容不支持继续编辑，或者编辑成本极高
3. **缺乏学术场景优化**——现有工具针对商业汇报，不适配学术论文的结构（背景 → 方法 → 实验 → 结论）
4. **PDF 解析质量差**——传统 PDF 解析工具无法正确处理双栏排版、公式、表格，提取内容顺序混乱

### 1.2 目标

打造一个**"论文 → 学术 PPT → 交互式局部修改"**的智能体系统，核心特性：

- ✅ **论文自动解析**——支持 arXiv LaTeX 源码 / PDF 输入，自动提取结构化内容
- ✅ **学术模板生成**——自动适配计算机论文汇报的标准结构（8-12 页）
- ✅ **局部交互修改**——点击选中元素 → AI 生成修改方案 → accept/reject（类似 Cursor）
- ✅ **多格式导出**——HTML 网页演示 / PPTX / PDF
- ✅ **公式/架构图渲染**——LaTeX 公式、Mermaid 架构图、实验表格

### 1.3 目标用户

- 计算机专业研究生/本科生（学术汇报、组会、毕业答辩）
- 后续扩展到其他理工科专业

## 三、核心数据流

### 3.1 数据格式定义

#### 中间格式（论文结构化内容，无视觉信息）

```json
{
  "paper_info": {
    "title": "论文标题",
    "authors": ["作者1", "作者2"],
    "venue": "会议/期刊名",
    "year": 2025,
    "abstract": "摘要内容（2-3句话）"
  },
  "problem": {
    "description": "要解决的核心问题",
    "existing_methods_limitations": [
      "现有方法1的不足",
      "现有方法2的不足"
    ]
  },
  "method": {
    "overview": "方法概述（3-5句话）",
    "framework_diagram": "Mermaid 格式的架构图描述",
    "modules": [
      {
        "name": "模块名称",
        "description": "模块说明（2-3句话）",
        "formulas": [
          {
            "latex": "公式 LaTeX 代码",
            "description": "公式含义说明"
          }
        ]
      }
    ]
  },
  "experiments": {
    "datasets": ["数据集1", "数据集2"],
    "baselines": ["基线方法1", "基线方法2"],
    "metrics": ["指标1", "指标2"],
    "results": [
      {
        "name": "实验1名称",
        "description": "实验说明",
        "table": "Markdown 格式的实验结果表格"
      }
    ],
    "ablation": [
      {
        "name": "消融实验说明",
        "table": "Markdown 格式"
      }
    ]
  },
  "conclusion": {
    "summary": "结论（2-3句话）",
    "future_work": "未来工作方向"
  },
  "references": [
    "参考文献1",
    "参考文献2"
  ]
}
```

#### PPTist 数据模型 JSON（页面结构，包含视觉信息）

每个元素是一个 JSON 对象，包含：
- `id`：唯一标识
- `type`：元素类型（text / image / shape / table / chart / formula）
- `left`, `top`, `width`, `height`：位置和尺寸
- `rotate`：旋转角度
- `defaultStyle`：默认样式（字体、颜色、大小等）
- `content`：内容（文字、LaTeX 公式、表格数据等）
- `slideId`：所属页面 ID

## 四、分阶段执行计划

### Phase 0：验证可行性（1-2 周）

**目标：** 确认"论文 → PPTist JSON → 渲染"这条路走得通

**任务清单：**
- [ ] **研究 PPTist 数据模型结构**
  - 克隆 PPTist 仓库：`pipipi-pikachu/PPTist`
  - 分析 `src/views/Editor.vue`、`src/types/slides.ts` 等核心文件
  - 理解幻灯片 JSON 的完整结构（pages、elements、theme 等）
  - 输出文档：`PPTist_数据模型分析.md`

- [ ] **手动创建测试数据**
  - 选 1-2 篇计算机论文（推荐 arXiv 上的，结构简单清晰）
  - 手动将论文内容转换为 PPTist JSON 格式（8-10 页）
  - 覆盖核心元素类型：标题、正文、公式、表格

- [ ] **验证渲染**
  - 将手动生成的 JSON 导入 PPTist，检查是否能正确渲染
  - 重点验证：公式（KaTeX 渲染）、表格、Mermaid 图表

- [ ] **输出物**
  - PPTist 数据模型分析文档
  - 1-2 份手写的测试 JSON 文件
  - 渲染效果截图

**验收标准：** 手动生成的 JSON 能被 PPTist 正确渲染，公式和表格正常显示。

---

### Phase 1：论文解析 Agent（2-3 周）

**目标：** 论文解析 Agent 能稳定输出结构化的中间格式

**任务清单：**
- [ ] **定义中间格式 JSON Schema**
  - 确定字段、类型、必填/选填
  - 输出文档：`中间格式_Schema.md`

- [ ] **LaTeX 源码解析**
  - 解析 `.tex` 文件结构：`\title`、`\author`、`\abstract`、`\section`、`\subsection`
  - 提取公式（`$...$`、`$$...$$`、`\begin{equation}`）
  - 提取表格（`\begin{table}`、`\begin{tabular}`）
  - 提取图片引用（`\includegraphics`、`\caption`）

- [ ] **设计 LLM Prompt（论文 → 中间格式）**
  - 系统提示词：角色定义 + 输出格式约束
  - 用户输入：LaTeX 源码（或 Markdown 文本）
  - 输出：中间格式 JSON
  - 版本管理：记录每次 Prompt 修改的测试效果

- [ ] **构建测试集**
  - 收集 5-10 篇计算机论文（覆盖不同子领域：CV、NLP、推荐系统等）
  - 手动标注每篇论文的标准中间格式（作为 ground truth）

- [ ] **评估提取准确率**
  - 逐字段对比 Agent 输出 vs 标准答案
  - 计算各字段的 F1 分数
  - 目标：核心字段（title、abstract、method、experiments）准确率 > 80%

- [ ] **（后续）MinerU PDF 解析集成**
  - 在 4060 电脑上安装 MinerU
  - 开发 PDF → Markdown 的解析管道
  - 与 LaTeX 解析输出统一为中间格式

**验收标准：** 10 篇测试论文的平均提取准确率 > 80%，核心字段无明显遗漏。

---

### Phase 2：演示文稿绘制 Agent（3-4 周）

**目标：** 输入中间格式，自动渲染出完整的学术 PPT

**任务清单：**
- [ ] **设计学术 PPT 模板（JSON 骨架）**
  - 标准 10 页模板：
    1. 封面（标题、作者、会议、日期）
    2. 研究背景 & 动机
    3. 研究问题
    4. 方法概述（架构图）
    5. 方法细节 1
    6. 方法细节 2（公式）
    7. 实验设置
    8. 实验结果（表格/对比图）
    9. 消融实验
    10. 结论 & Q&A
  - 定义每页的布局类型：单列、双列、三列、全图
  - 输出文档：`学术PPT模板设计.md`

- [ ] **设计 LLM Prompt（中间格式 → PPTist JSON）**
  - 系统提示词：模板规则 + 元素布局约束
  - 用户输入：中间格式 JSON
  - 输出：PPTist JSON 数据模型
  - 重点约束：
    - 公式使用 KaTeX 格式
    - 架构图使用 Mermaid 格式
    - 表格使用 Markdown 格式
    - 每页字数上限控制

- [ ] **KaTeX 公式渲染集成**
  - 在 PPTist 中添加公式元素类型
  - 支持 LaTeX 公式的实时渲染
  - 测试常见公式：数学符号、矩阵、分式、上下标

- [ ] **Mermaid 架构图渲染集成**
  - 在 PPTist 中添加图表元素类型
  - 支持 Mermaid 语法渲染（流程图、序列图、甘特图等）
  - 测试常见图：模型架构图、训练流程图

- [ ] **实验表格渲染**
  - 在 PPTist 中支持 Markdown 表格 → HTML 表格渲染
  - 支持表格样式（表头加粗、奇偶行交替颜色等）

- [ ] **端到端测试**
  - 输入 5 篇论文的中间格式
  - 自动生成 PPTist JSON 并渲染
  - 评估：内容完整性、排版合理性、视觉美观度

**验收标准：** 5 篇测试论文自动生成 PPT，内容完整、排版合理，公式/表格/图表正常渲染。

---

### Phase 3：修改 Agent（3-4 周）

**目标：** 支持局部修改，accept/reject 交互流畅

**任务清单：**
- [ ] **元素选择交互**
  - 用户在 PPTist 中点击选中元素
  - 获取该元素的 JSON 数据（id、type、style、content 等）
  - 显示元素信息面板（便于调试）

- [ ] **修改指令输入**
  - 侧边栏输入框，支持自然语言指令
  - 示例："把这个标题换成更大的字体"、"把这个表格换成柱状图"

- [ ] **设计 LLM Prompt（理解修改意图 + 生成修改方案）**
  - 系统提示词：修改规则 + 全局风格约束
  - 用户输入：
    - 当前元素 JSON
    - 全局样式配置（配色、字体等）
    - 用户修改指令
  - 输出：修改后的元素 JSON（diff）

- [ ] **修改预览 & accept/reject**
  - 在侧边栏显示修改预览（当前 vs 修改后）
  - 用户点击"接受"应用修改，或"拒绝"撤销
  - 支持多轮微调（"再大一点"、"颜色再浅一点"）

- [ ] **全局风格约束**
  - 修改 Agent 需要读取全局样式配置：
    - 主题色
    - 字体家族
    - 字号层级
    - 间距规范
  - 确保局部修改不破坏整体风格

- [ ] **修改历史记录**
  - 记录每次修改操作（元素 ID、修改内容、accept/reject）
  - 支持 undo/redo
  - 防止修改冲突（如两个用户同时修改同一元素）

- [ ] **端到端测试**
  - 5 位同学试用，每人修改 1 份生成的 PPT
  - 收集反馈：定位准确性、修改质量、交互流畅度

**验收标准：** 5 位用户试用，局部修改成功率 > 70%，accept/reject 交互流畅。

## 六、Prompt 工程体系

### 6.1 Prompt 设计原则

1. **每个 Agent 独立维护 Prompt 文件**——不混写，便于版本管理
2. **Prompt 版本化**——每次修改 Prompt 都要跑测试用例，记录效果变化
3. **系统提示词 + 用户提示词分离**——系统提示词定义角色和规则，用户提示词传入具体数据
4. **输出格式强约束**——要求 LLM 输出 JSON，用 JSON Schema 校验

### 6.2 Prompt 文件结构

```
prompts/
├── paper_parser/
│   ├── system_prompt_v1.md    # 论文解析 Agent 系统提示词 v1
│   ├── user_prompt_template.md
│   └── test_results/           # 每次测试的结果记录
├── ppt_generator/
│   ├── system_prompt_v1.md
│   ├── user_prompt_template.md
│   └── test_results/
├── modifier/
│   ├── system_prompt_v1.md
│   ├── user_prompt_template.md
│   └── test_results/
└── global_style/
    └── style_config.json       # 全局样式配置
```

## 七、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| **PPTist 数据模型过于复杂** | Phase 0 延期 | 先用最简子集（text + image），后续逐步扩展 |
| **LLM 输出 JSON 格式不稳定** | 各阶段受影响 | 加 JSON Schema 校验 + 自动重试机制 |
| **MinerU 部署困难** | Phase 1 的 PDF 解析卡住 | 先用 LaTeX 源码替代，MinerU 作为后续优化 |
| **公式渲染兼容性** | 复杂公式渲染异常 | 先用 KaTeX 支持的基础公式，复杂公式降级为图片 |
| **局部修改定位不准确** | 修改 Agent 用户体验差 | 强制用户先点击选中元素，再输入修改指令 |
